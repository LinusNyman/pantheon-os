## 5. The Pantheon spine

Pantheon is a library (linked by everything); its `pan` binary — the structural CLI (§5.5) and TUI (§10) — is a thin separate crate over it (§14), so the spine lib itself carries no UI deps. The library owns addressing, resolution, the record envelope, the `Core` substrate (§7.1), and write-time validation. Nothing else stores state on its behalf.

### 5.0 The tree is the ontology

There is no `ontology.toml`. Everything about a node is read off its directory:

- **existence** — the directory is there
- **code, char, label** — its name (the triple, §5.1)
- **parent, children** — its position

A node's meta file (`[code]__.toml`, §5.2) carries only what the name can't: symbol, keywords, deity, explanation. It annotates, never defines, and is optional — mainly the top ontological layers carry one.

**Resolution is a walk, not a cache.** There is no persistent index and no `pan reindex`. Every address resolves by walking the tree on demand, into a **transient in-memory map built once per command** (or once per data-refresh in a TUI), then discarded:

- **code → path** is derived by descending the tree, one directory globbed per level (§5.1); no map needed — three `readdir`s find `csa`. A compact code tokenizes from the string alone; a definition-prefix code (§5.1) does not, and is matched level-by-level against the tree instead — still one directory per level.
- **`core:slug` → path** is a walk of that core's files only (`*__person__*.json` and `*__person.json` for Album's `person` kind). A partitioned entity is one file, so resolution finds it by filename without opening it (§5.2): its slug is the file's last `__`-segment (`csa__person__john_appleseed.json`) or, for an **entity-as-node**, the node's own definition (`csa_john_appleseed__person.json` → slug `john_appleseed`, §5.1). A **named series** resolves the same way — its name is the file's last `__`-segment (`cso__log__meetings.jsonl` → `meetings`), matched by filename without opening it; only a *name-keyed line* (a Pensum task) needs the series opened and its keys scanned — a tree-wide read of every `task.jsonl` in scope, the one resolution that opens record files rather than resting on filenames (still one walk into the map per command, and cheap at personal scale). The naming triple isolates the walk to the small text, bulk is never opened (§6.3). A command resolving many refs walks once into the map, then does in-memory lookups.

  *Which files are "that core's"?* Kinds are **globally unique across cores** (`person`/`organization` are Album's alone), so a file's `[kind]` segment names its owning core, and the meta dir is the only place these files live. The spine learns each installed core's `kinds()` by **runtime PATH discovery** (the same discovery behind `pan doctor`, §5.5) — never by linking a core (I5) — and so filters the walk to Album's kinds and enforces per-core slug uniqueness (§5.4) without importing Album.

This is I6 taken to its end: the tree is the only state, with nothing cached beneath it. Nothing to reindex, invalidate, or keep out of sync — a hand reorganizing the tree with `mv` is immediately visible to the next command, because the next command re-walks. The walk descends only into node dirs — triple or definition-prefix (§5.1) — and their meta dirs, so bulk content is never scanned.

The one cost is a cold scripted loop of many independent resolves paying one walk each; `pan resolve` accepts multiple refs and walks once, and a long-running process holds its map — so at personal scale a resolve is sub-100 ms and no cache earns its keep.

### 5.1 The naming triple

A node directory is named **`[parent code]_[defining char]_[definition]`**:

- `a_actio` — root node: no parent code, char `a`, label `actio`. Full code `a`.
- `a_c_cura` — parent `a`, char `c`, label `cura`. Full code `ac`.
- `cs_a_amicitia` → `csa`; `ec_v_valetudo` → `ecv`.

A node's full code = parent's code + its defining char, so any code reconstructs the path and vice versa. Files inherit the full code as prefix: `ecv_doctors_appointment.md`, `csa__person__alex.json`.

**A node without a defining char — the definition-prefix form.** Some nodes have no natural one-character char: a contact promoted to their own node (`john_appleseed`) would need an initial that collides with a sibling or a number that means nothing. Such a node is instead named **`[parent code]_[definition]_`** — no char slot, the normalized definition kept whole (its internal `_` intact), and a single **trailing `_`** as a marker. Its full code is `[parent code]_[definition]` (the trailing `_` is a directory device, not part of the code), and the definition **doubles as the label**. Files inherit that code as prefix exactly as a triple node's do — `csa_john_appleseed_/` (code `csa_john_appleseed`, label `john_appleseed`) holds a loose doc `csa_john_appleseed_example.md` and its own record `csa_john_appleseed__person.json` (§5.2).

The trailing `_` is what tells the two forms apart at read time. `aook_a_first_thing` (no trailing `_`) is a triple — char `a`, code `aooka`; `csa_a_name_starting_on_just_one_letter_` (trailing `_`) is definition-prefix — code `csa_a_name_starting_on_just_one_letter`. A single-letter char and a single-letter *definition* thus stay distinct (`csax` vs `csa_x`), and neither collides with the meta dir, which ends in `__` rather than a single `_` (§5.2). A definition-prefix node's **children may themselves only be definition-prefix** — a one-char triple char can't be concatenated onto an `_`-bearing code unambiguously — so `csa_john_appleseed_project_x_/` is valid (code `csa_john_appleseed_project_x`), each level adding one `_`-joined definition segment.

Two naming invariants keep a **compact** (all-triple) code string unambiguous *by construction*, so it parses left-to-right with no lookup:

- **A letter char is exactly one character**, unique among its siblings — usually the label's first letter, free to differ on clash (`csng` Gens vs. `csnf` Familia).
- **A numeric level uses a fixed width of 2 digits** (`01`…`99`), so every numeric token is exactly two characters and codes sort correctly as text. Such a level takes a defining char above it — `asd_l_lecture`, then `asdl_01_…`, `asdl_02_…` — so the char slot holds one alphabetic character or two digits and never one digit, and a digit never opens a code. A level never needs a 100th sibling: if it seems to, that level wanted splitting (I7).

A scanner then reads a compact code deterministically: a letter is a one-char token, a digit begins a two-digit token. `asdl10` → `a, s, d, l, 10`; two adjacent numeric levels `…0103` → `01, 03`. Each token globs `{parent-code}_{char}_*` at its level and matches exactly one directory (the char is unique among siblings). No ambiguity, no index.

A **definition-prefix segment cannot be tokenized from the string alone** — its internal `_` (from folded spaces) are indistinguishable from the boundaries between stacked segments — so a code carrying one is resolved not by scanning the string but by **one downward walk**: at each level exactly one child directory's code is a prefix of the remaining string (`csa` → `csa_john_appleseed_` → `csa_john_appleseed_project_x_`), and the code-collision check (§5.3, §5.5) guarantees that match is unique. This is still a single tree pass with no index: both forms descend one directory per level onto exactly one node — the compact case can pre-tokenize from the string with no disk read, the definition-prefix case does its tokenizing by the walk itself.

**Name normalization — one rule, every typed token.** A code char, a label, a slug, a kind, a series or rule name, a filename: each is normalized to a single legal form on the way in, so the same input always yields the same name and any hand's typing lands identically (I8). The form is:

- **lowercase**, **NFC** Unicode;
- keep Unicode **alphabetic or numeric** characters and `_` — so `å ä ö` survive, and so do the combining vowel marks Devanagari, Thai, and Arabic spell words with (a letters-only cut would drop those, leaving a different word standing; *alphabetic* is the Unicode property, wider than the letter categories, and is what std's `char::is_alphanumeric` tests, §13);
- fold **space** and **`-`** to `_`; drop every other punctuation/symbol;
- **collapse runs** of `_` to one; **strip** leading/trailing `_`; **reject** a token that normalizes to empty.

NFC is not optional: macOS and Linux disagree on decomposed vs. composed forms (`ö` as one codepoint, or `o` + combining diaeresis) — two names that read identically yet differ as bytes, so a ref would miss its file. Normalizing on write and comparing NFC in resolve/validate closes that gap. `__` is reserved as the file-field separator (§5.2); the collapse-and-strip rule guarantees a normalized token can never contain it. A directory or filename **composes** normalized tokens with structural separators — the triple's `_`, the definition-prefix's trailing `_`, the meta dir's `__` — which sit *between or around* tokens, never inside one, so the strip-trailing-`_` rule (which governs each token) and these markers never conflict: a definition token like `john_appleseed` is stripped and collapsed, then the trailing `_` is added on composition. Normalization is deterministic, total, and idempotent — one input, one output, `normalize(normalize(x)) == normalize(x)` — which is exactly what lets `pan` auto-apply it as a fix (§10.2).

### 5.2 The meta dir

Each node owns one meta dir, `[full code]__` (the char/definition slots empty), holding the node's machinery — a triple node `csa` has `csa__`, an entity-node `csa_john_appleseed` has `csa_john_appleseed__`. Every record file is named so that a walker can classify it **without opening it or asking a core** — the **extension** tells the storage shapes (§6.1) apart:

- `[code]__.toml` — node annotations: symbol, keywords (for LLMs), deity/personification (a memory aid), explanation. Optional; mainly the top layers carry one.
- `[code]__[kind]__[slug].json` — a **partitioned register**: one object per entity, kind first and slug last (`csa__person__john_appleseed.json`, `cso__organization__dare_robotics.json`). Album, Mappa, Rationes.
- `[code]__[kind].json` — an **entity-as-node**: the self-record of a definition-prefix node, whose slug *is* the node's definition, so the filename carries only the kind (`csa_john_appleseed__person.json`, slug `john_appleseed`). The entity that would otherwise be `csa__person__john_appleseed.json` up in the parent's meta dir instead lives at, and is named for, its own node (§5.1). One per node.
- `[code]__[kind]__[name].jsonl` — a **series**: one file per collection, many keyed lines (`cso__log__meetings.jsonl`, `cso__event__standups.jsonl`); nameless as `[code]__[kind].jsonl` where the core's series is determined rather than hand-named (`cso__task.jsonl`, §7.3). Annales, Fasti events, Pensum, Rationes balances.

**The kind is always the first `__`-segment after the code**, and the slug/name (when present) is last; a form lacking that trailing slug/name — `[code]__[kind]` (an entity-node or a nameless series) or `[code]__` (the annotation, an empty kind) — reads the same way. All cores' files live in their node's meta dir — the `__` after a code marks a meta-dir record — **except Tabella's** documents, which stay loose in the open node dir (§6.1).

**The parse rule** (structural, no per-core knowledge): extension alone classifies the *shape* — `.toml` is the node's annotations, a document extension (`.md`, `.txt`, `.mdx` — a small fixed set) a document, `.json` a partitioned entity, `.jsonl` a series — then splitting the stem on `__` gives `[code, kind, name?]`: code first, **kind always second**, an optional slug/name last (absent for an entity-node `[code]__[kind].json` or a nameless series `[code]__[kind].jsonl`). The code segment may carry single `_` from a definition-prefix node but never `__`, so the split is clean (§5.1). The `[kind]` vocabulary is declared by each core, which validates its own filename segment on write (§7.1, where the tokens are listed once); the walker needs only the extension and the split, never a token's meaning. Auspex rules share this meta-dir shape — `[code]__function__[name].ext`, the reserved **`function`** token in the kind slot (§9.1) — so they too are double-underscore names living in the meta dir. A rule's extension is a *language* and may be absent entirely (the shebang naming it — `crp__function__balance_check`), so the walker recognizes a rule by that reserved segment rather than by extension, and checks the segment **first**: a `function` file wearing a document's extension is still a rule, and the two classifiers never race.

Child-node dirs and loose `[code]_*` documents (`.md`/`.txt`/`.mdx`) stay in the open node dir.

### 5.3 Code collisions

A **code collision** — two nodes resolving to the same full code, or one whose code prefix-shadows another's walk — is the only ambiguity §5.1's rules admit, and it arises only from a naming mistake: a reused char among siblings, or a definition-prefix that prefix-shadows a sibling lineage (a node `csa_john` beside a node `csa_john_appleseed`, where the shorter would divert the longer's walk). **`pan new` refuses to mint one**, the check being a glob of the parent's children — and the `--def` form pays it too, since §5.4's own nudge for a duplicate slug (a second `alex` to `alex_smith`) manufactures exactly this shadow where both Alexes are entity-as-nodes, prescribing the collision it warns about. `pan validate` then reports by path what a hand's `mkdir` minted behind `pan`'s back, which is the only way one still arrives (I8).

### 5.4 References & the record envelope

One reference form to another record, plus one for bulk (§6.5):

- **Entity, document, or hand-named series** → `core:slug` (`album:alex`, `tabella:trip_idea`, `annales:meetings`) — the three storage shapes (§6.1), each addressed by a per-core-unique token. A **hand-named series** — an Annales log, a Fasti event collection (§7.3) — is referenced as a whole by its series name and resolved by filename like an entity (`fasti:standups`); a *date-keyed line* within it is a sample, not an identity, and is never a target (I1), whereas a *name-keyed* line (a Pensum task) is, reached by its key (`pensum:reach_out_to_alex`). A determined-name series (a Rationes `balance`, declared `Series { named: false }`, §7.1) is reached through its entity, never referenced on its own: its name slot carries its determinant, not an identity, so it is no more a ref target than a `[kind]` segment is, and its uniqueness is its determinant's — which is why the spine reads `named` to know, without a token's meaning, that this name is not one to check. Every ref is resolved by walking the core's files (§5.0) and survives a re-home untouched, since the token carries no path. A **rename** changes a slug (a name and its slug are one thing, below) — as does renaming a definition-prefix node, whose slug *is* its definition (§10.1); both cascade, so no ref is left dangling. An identifier — an entity or document slug, a task key, or a hand-named series's name — must be unique **per core** for a ref to resolve to one record. All of them are read off filenames (§5.0), so they fall into one per-core uniqueness set, checked where the check is cheap. **Within a node** it is one `readdir`: the core globs its own meta dir and `add` refuses a slug another of its kinds already holds (exit `3`, §18) — `csa__person__book_club.json` and `csa__group__book_club.json` differ as files and collide as refs, so the filesystem permits what the ref namespace does not. **Across nodes** it is a walk, so it stays soft — `pan validate` reports it and `add` warns, nudging a second `alex` to a fuller name (e.g. `alex_smith`) — and a resolve meeting two candidates lists them rather than guessing (§7.3). A document behaves identically: its filename slug is its name, and its cross-node duplicates are the same soft finding.

No *record* references a node. A record about a node lives in it (I3); aboutness is location, so a node is never something a record points at, and a ref therefore never encodes a home — there is nothing to falsify on a re-home. One code-valued token on disk does name a node, and it is not a record's: a rule's `writes=core@home` capability (§9.2) grants authority at a node rather than pointing at one, and it survives a re-home the only way a code can — `pan rename`/`mv` cascade it, `pan validate` reports a dead one (§10.1–§10.2). Legibility is a render concern: tables, TUIs, and `pan resolve` display the entity's current home for context, e.g. `album:alex (csa)`.

A record's `home` and `core`, and — for a partitioned entity — its `kind` and `slug`, are its file's location and name, never stored, so they cannot drift (I3). For an **entity-as-node** the slug is the node's definition rather than a filename tail (`csa_john_appleseed__person.json` → `john_appleseed`, §5.2) — still part of its name, still unstored. The record body then takes one of two JSON on-disk forms — a partitioned entity object or a series line — while a document, the third shape (§6.1), carries its metadata as TOML frontmatter over a prose body instead (§6.6).

A **partitioned entity** file (`csa__person__john_appleseed.json`, or `csa_john_appleseed__person.json` for an entity-as-node — same object, slug taken from the node's definition) is just the entity's object — `kind` and `slug` are the name, not written inside:

```json
{ "refs": ["album:mara"],
  "data": { "away": [{"from":"260601","to":"260615"}], "…": "…" } }
```

A **series line** (one line of a `.jsonl`) carries the per-record envelope, since the key varies line to line:

```json
{ "key":"260703", "refs":[], "data":{"…":"…"} }
```

`key` — the record's identity within its series and the target for `edit`/`rm`: a date (`YYMMDD`, plus a time when a hand gives one, §7.3) for a reading, or the record's name for a register line (a task). A name is normalized to its slug on the way in (§5.1), never hand-typed as one. A **date key is stable** — the reading's own identity, unchanged on edit. A **name is its slug**: correcting it (`johnn` → `john`) renames the record and **cascades every `core:slug` ref** pointing at it (`rename`, §7.2), so a fixed typo is total and orphans nothing — a name and its slug are never allowed to differ. Because there is no reverse index (§18), the cascade finds those refs with one tree walk (affordable at personal scale, §5.0), rewrites them as a planned transaction (non-atomic; a crash mid-cascade is diagnosed from the tree by `pan validate`, §10.1), and batches the writes so a single, **triggerless** `aus run` hook fires — one wake, and no write for it to name where the batch touched several cores and homes at once (§9.3–§9.4). The ref rewrite edits only the envelope's raw `refs` array, which Pantheon owns — so it is a spine operation and no core touches another core (I5). `refs` — `core:slug` links, resolved lazily. Nothing records who wrote a line: a rule is code you authored and granted, so what Auspex writes is your own hand at one remove (I8, §9.5), not a foreign one to be marked.

### 5.5 `pan` CLI

- `pan tree [code]` — emit the ontology (sub)tree as JSON, derived from the directories.
- `pan resolve album:alex [more:refs…]` — locate and dereference entity refs; accepts many and walks once (§5.0).
- `pan csa` — code ↔ path, label, symbol, keywords.
- `pan cd <code>` — emit a node's absolute path for a shell shim to `cd` into: `pan cd ao` walks from the root (§5.1) and prints Opus's dir; a `core:slug` works too, landing at the entity's current home node (§5.4). A process can't change its parent shell's cwd, so `pan init <bash|zsh|fish>` emits a `pan` wrapper that performs the `cd` (the zoxide pattern) and passes every other verb through untouched — the same code→path resolution the other hands use pathlessly (I8). You land in the open node dir, so `$PWD` reads back as that code, the inverse of the locus walk (§7.3). Bare `pan cd` (no code) lands at `$PANTHEON_ROOT` itself. No such node → exit `4`; nothing is stored, it moves the real `$PWD` (no cursor, §7.3).
- `pan constitution [code]` — emit the seven placement rules (§2) plus a node's keywords, so an LLM files as well as a human.
- `pan doctor` — installed apps, their crate and format versions, mismatches (§15.5), and any collision across two cores' declared tokens. Versions come off each app's `version -f json` (§7.3), a verb every layer has; tokens come off each **core**'s `schema` (§7.1–§7.2), since with in-record kinds gone every filename token flows from that one surface. The split is what lets `doctor` see all twelve: a lens owns no records and so declares no `schema` (§12), and reading versions off that surface would blind the check to the very apps §15.5 says installing one of makes real. A clean run means the file→core map is **total**: every token has exactly one owning core and one shape, which is what lets resolution read a name without importing a core (§5.0).
- `pan migrate` — rewrite the tree from one format version to the next. **Shape-directed and idempotent**: it rewrites the old forms it meets and passes over the new, so it is run rather than aimed (§15.5). Same planned transaction and dry-run-first as the structural operations below (§10.1) — and a crash mid-migration is answered by running it again, that idempotence being the whole of its recovery story, since nothing on disk stamps how far it got.
- `pan validate` — the tree's own lint, findings reported by path and enumerated once, at §10.2. Whether a filename's kind segment is *legal for its owning core* is that core's check (§7.1), not `pan`'s — the spine links no core (I5); `pan` flags only a token owned by no installed core — an unclassifiable file (§5.0, §7.1). This is the lint that re-checks hand-edited files (I6); it runs **on demand** — nothing watches the tree (§18) and `aus run` walks for rules, not findings (§9.4), so a hand edit stands until you look, or until the owning core reads the file and refuses what won't deserialize (exit `3`, §13).
- `pan annotate <code> [--set key=val …]` — read or edit a node's `[code]__.toml` meta file (symbol, keywords, deity, explanation, §5.2), written in place via `toml_edit` so hand comments and ordering survive (§6.6). With no `--set` it emits the current annotations as JSON; annotate touches the node, never `data`, and never the tree shape. The CLI parity of the §10.3 TUI action, so an LLM or script annotates as fluently as a hand (I8).
- `pan mv-file <file> --to <code>` — re-home one file to another node, rewriting its `[code]` prefix: a misfiled record, or a whole series (§7.2). Same planned transaction and confirm as the node operations below; it moves a file, never a node.
- `pan new <parent> <char> <label>` / `pan rename <code> [--char C] [--label L] [--def D]` / `pan mv <code> --to <parent>` / `pan rm <code>` — structural operations, always as a dry-run-first planned transaction (§10.1). `pan new` accepts a **batch** — several siblings minted in one plan, chars auto-assigned unique among siblings (§5.1), annotated afterward (`pan annotate`); a node with no natural char is minted as a **definition-prefix** node instead (`pan new <parent> --def <definition>`, §5.1), whose children must likewise be definition-prefix; mint only nodes the reality test admits (I7), never speculative blanks. **A sphere is minted against `root`** — `pan new root a actio` mints `a_actio/`, a root node having no parent code to name (§5.1), so `root` is the one parent token that is not a code: reserved like a verb, and warned on by `pan validate` where the ontology mints a code colliding with it (§7.3). The tools ship no spheres and urge none (§2); `root` is only how a hand or an LLM mints its own where `mkdir` would (I8).
- `pan rename-prefix <old> <new> [code]` / `pan rename-pattern <from> <to> [code]` — **bulk name fixes over a subtree you ask for** (the request *is* the intent, so it is unambiguous and `pan` applies it, unlike a `validate` finding it must guess at, §10.2): rewrite one code-prefix to another, or substitute a literal across dir labels, doc filenames, slugs, and series names (`youtube` → `yt`). A substitution landing on a slug or a named-series name **cascades its refs** (§5.4). Same planned transaction, same normalization (§5.1), same confirm (§7.3).
