## 16. Build order

A vertical slice before breadth, so the contract is proven end-to-end early. Cores land **CLI-first** — a bare short prints `help` until its TUI exists (§7.3) — so the chrome arriving mid-list is what the grammar already allows, not a debt it runs up. The repo is public and CI green from the first commit: licences, hygiene, and the per-PR checks (§15.1–§15.2) precede this list rather than closing it, `dist plan` alone excepted — dist ignores a package that defines no binary, so a workspace of libraries gives it nothing to plan and that check joins at step 1 (§15.2).

1. **Pantheon + `pan`** — the spine (addressing, resolution, the record envelope, the `Core` substrate, write-time validation — §5) and the thin bin over it (§5.5, §14). The spine is a library, so nothing exercises it until its bin exists: `pan new` mints the tree a core will later write into (§7.3), and `pan tree` / `resolve` / `validate` are how it is read back before any core can. First binary in the workspace, so `dist-workspace.toml` and CI's `dist plan` land here (§15.2–§15.3).
2. **Annales** — the simplest core, one hand-named `log` series (§8.6); proves I1, the Series shape, and the contract.
3. **Album** — the first partitioned register (§8.1); proves the second shape, `core:slug` refs and the resolver's filename path (§5.0), and the rename cascade (§5.4).
4. **Pensum** — the nameless `task` series (§8.5), minted by a node's first task rather than by `-c`: the determined-name path Annales' hand-named log never reaches (§7.3). It is also the file a detached hook and a hand contend for, which is what the record lock exists for (§6.4), and the tasks Atrium's day is made of (§12).
5. **Tabella** — the third shape (§6.1, §8.7). A Document core declares no tokens at all (§7.1), so it is the one file→core mapping resting on extension alone, and the one that makes `pan doctor`'s totality claim demonstrable rather than merely stated (§5.5). Its `edit` is the editor form's home ground (§7.3).
6. **Porticus + Tessera + Atrium** — the first screen; proves derived-out rendering (I1) and the relayed human write (I2, §12), and gives the cores and `pan` their TUIs at once. The chrome's design is `PORTICUS-SPEC.md`'s (§11.1). The slice closes here: three shapes below, one contract across them, one screen above.
7. **Mappa, Rationes, Fasti** — the cores the slice did not need, against the proven contract. Fasti brings the two-token core (a partitioned `span` beside an `event` series, §8.4), Rationes the determined-name series (§8.3).
8. **Auspex** — the first rule, `plan` before `run`; proves the propose protocol, the capability grant, and the one reactive writer (I2). It lands last of the writers because a grant's vocabulary is `core@home` (§9.2): with every core built, the capability surface is written once against the whole of it rather than revisited as each target appears. Steps 1–7 are themselves the `aus`-not-on-`PATH` case (§9.4), so the hook spawn every core carries is exercised as a no-op long before there is anything to wake.
9. **Cleanups** — the deferrals steps 1–8 named and left, paid before the last breadth and the first release. Not new capability: each is a debt the slice took on deliberately, each traces to a spec section and a stub already in the tree, and each is small beside the step that deferred it — which is why they gather here rather than interrupting the vertical slice. Ordered largest first:
   - **`pan`'s node-level cascade (§10.1)** — the six structural mutators (`mv`, `rm`, `rename`, `rename-prefix`, `rename-pattern`, `mv-file`) return not-implemented, so `r`/`m`/`x` are dark in `pan`'s TUI. A node code *is* its path (§5.2), so a rename rewrites every child directory name and every `[code]` filename prefix under the branch, and cascades the definition-prefix `core:slug` refs and the `writes=core@home` grants it names (§9.2, §10.2) — one dry-run-first planned transaction, the record-level cascade (§5.4) already proven, now over the tree. A crash mid-cascade is diagnosed from the current tree by `pan validate`, never from an in-flight plan (§18). The largest of the debts and the one a hand asks for first.
   - **`pan migrate` (§5.5, §15.5)** — `Cmd::Migrate` is stubbed. Shape-directed and idempotent, it can only be built once a *prior* format version exists to rewrite from, so it lands here as the first version boundary approaches (step 11) rather than before — the one cleanup that is blocked, not merely deferred.
   - **`pan validate`'s auto-apply and candidate fixes (§10.2)** — a `Finding` today carries a code, a severity, a path, and a message, and no candidates; the validate tab shows findings it cannot offer to fix. This step makes the spine produce candidates: the single-legal-correction cases (case, NFC, a stray `-` or space in a name — §5.1) normalize directly and still confirm (§7.3), while a genuine choice surfaces candidate commands and the hand picks — the split §10.2 already specifies.
   - **The figlet `roman` banner (P§8)** — the Title overlay ships the tracked name-word in place of the figlet face, which needs a vendored third-party `.flf` that `cargo deny` cannot see. Settled here as the licensing call it is, not the coding one.
   - **`Pick` as P§4's tree-as-modal** — today a line prompt that resolves a home by typed code; the tree-as-modal is the nicer form of the same question, and the place a Full view's `a` earns it.
   - **Nested `data` render (§7.3)** — the table renderer prints a nested object raw inside its cell (Mappa's `{"lat":…,"lon":…}`); a fallback to pretty JSON, as `pan tree` does for a nested node, is the deliberate call left open at step 7. A spine commit, so it rides no core.

   And the chrome debts the first hands-on use of the screens surfaced (P§4–P§7) — none a test caught, because the slice only ever exercised a *flag* write and one terminal size:
   - **The add form has no screen (§7.3, P§7)** — the largest of these. `a` on an entity core relays `add -H <node>` with no name and no fields, and the chrome has only single-line prompts (`Overlay::Line`), so nothing collects them: `a` gives no prompt and mints nothing usable (seen trying to create a Mappa place). The CLI's editor form (`add`/`edit`) has no TUI equivalent and cannot open an `$EDITOR` down a relay pipe regardless. `add` wants at least a name and the record's fields — a multi-field form overlay, or a name prompt handing off to `edit`. The slice proved only a *flag* write (`pen edit --done`), so this was never reached; today it is a defect wherever `a` is offered.
   - **Responsive stack for narrow terminals (P§6)** — the Rail layout is a fixed `Horizontal` 34/66 split; below some width the rail and content should stack top over bottom rather than side by side. Derived from terminal size, so no knob (§18).
   - **A narrower rail on wide terminals (P§6)** — the same 34% is too wide on a large screen; the left pane should take a smaller share (or a capped column) as width grows. Still no config.
   - **A passive overlay yields to a navigation key (P§4)** — with Title (`+`) or Help (`?`) up, every key routes to the overlay handler, which only pops on `Esc`, so a view-switch or another chrome key does nothing until the overlay is closed first. A *passive* overlay (not text entry) should dismiss on such a key and let it through in one press.
   - **The header names the node, not the trail (P§4)** — `breadcrumb` joins every ancestor `code·label` from the sphere down (`c·contextus  cl·locus`); the current node alone is enough (`locus`). A header-format call, so confirm P§4's intent before trimming.
10. **Speculum, Studium** — the lenses whose reach is the point (§12): both compose across cores, and Studium's GPA folds Fasti spans (§8.4), so neither is buildable before step 7.
11. **Releases** — `release-plz`, the per-crate tags, and the first tagged `dist` run (§15.3–§15.4), once an app is worth installing.
