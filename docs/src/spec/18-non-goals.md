## 18. Non-goals — deliberately absent, do not add

Every entry below is something the system could obviously grow and deliberately hasn't. The reason rides with the prohibition, so a later hand weighs what adding it would cost rather than rediscovering why it is gone.

- **No daemon, no filesystem watcher.** Hooks wake Auspex (§9.4); a hand edit is caught by the owning core when it next reads the file (exit `3`, §13) and by `pan validate` on demand (§5.5), never by something watching. Do not add a background process.
- **No audit log, no undo layer, no version history, no transaction history, and no VCS the tools keep over the records.** `edit` rewrites the keyed record in place, keeping no prior copy (§6.4); what is historic is intrinsic — a series' one-line-per-date trend, and list-valued fields on entities; no record carries its author, and a rule that logs its own runs writes an ordinary Annales log you chose to keep, never a sink the system reads (§8.6, §9.5); disaster recovery is backups and snapshots. `pan`'s structural operations keep no in-flight plan — a crashed rename is diagnosed from the current tree via `pan validate` (§10.1). A repo of your own at a project node is bulk like any other (§6.5, §13); what the tools keep is nothing.
- **No central store, no database, no opaque IDs.** Records are plain files at their nodes, keyed by human tokens (I6).
- **No persistent index, no `pan reindex`.** Resolution walks the tree on demand into a transient per-command map (§5.0). Codes resolve to exactly one node by construction (§5.1) — compact codes parse standalone, definition-prefix codes by a one-pass walk with a unique match per level; slugs resolve by a records-only walk. Do not add a cached index — the tree is the only state.
- **Nothing hidden, and nothing outside the tree.** No `.pan/` or other dotfile store, no OS cache dir, and no root marker; nothing sits at the tree root but the ontology's top nodes, and nothing on disk stamps a format version — which is why `pan migrate` is shape-directed rather than aimed (§15.5), and why the tree is its own version as it is its own ontology. Every byte the system relies on sits in the tree at its node, legible. Advisory write-locks (§6.4) are taken on the record files themselves, not on a side store; there is no process scratch kept anywhere off the tree.
- **No stored derived values.** No `current_*` fields, no cached tile values, no denormalized presents (I1). No stored `kind` or `home` anywhere — kind is the filename segment, home is the path, and a core whose `Record` is an enum over its tokens writes no variant tag, the filename having already named the variant (§7.1). A partitioned entity stores no key either — its *name* is the key (§5.2, §5.4) — while a series line carries the `key` its lines are told apart by, many lines sharing one filename (I3).
- **No auto-created containers.** A plain `add` mints no container — not a node (that is `pan new`) and not a hand-named series (that is `add -c`, §7.3); the entity or document it creates *is* the record, never a vessel for later ones. A hand-named series that resolves to nothing is a not-found error (exit `4`), never a new empty log conjured from a typo; inference (`ann weight …`, `ann ecv …`) only ever *finds* an existing series, and reports a conflict rather than guessing when more than one matches. (A determined-name series — Pensum's fixed `task`, Rationes' per-holding `balance` — is minted by its determinant, §7.3, not conjured by a guess.) Do not let a *hand's* write bring its own container into being. **Auspex is the one exception, and its grant is what bounds it**: a proposal's `add` mints a series only where the capability *names* that series (`annales@csa/stale_contact:add`, §9.2), so it is authored once — in the header, where you read it before granting — and a rule can no more conjure a log per run than a hand can conjure one per typo. A bare `core@home` grant appends and never mints.
- **No hard *global* uniqueness validator.** A slug names one record per core, and is checked where the check is cheap and nowhere else. *Within a node* that is one `readdir`, so `add` **refuses** a slug another of the core's kinds already holds (exit `3`): the filesystem refuses a duplicate *filename*, which is not the same guarantee — `csa__person__book_club.json` and `csa__group__book_club.json` are two files and one ref (§5.4, §6.1). *Across nodes* it is a tree walk, so it stays **soft**: `pan validate` reports it, `add` warns, a resolve meeting two lists them (§7.3), and you fix it at the source. Do not make the cross-node check hard — the walk is the cost the softness exists to avoid — and do not let the within-node one grow past its `readdir`.
- **No stored `home`, and no node reference in a record.** The path is the home (I3); a record about a node lives in it rather than pointing at it. `core:slug` is the only reference form a record has (plus relative paths for bulk, §6.5). Do not add a code-valued ref *to a record* — it would encode a location that a re-home falsifies. **One code-valued token exists and is not a record's:** a rule's `writes=core@home[/series]` capability (§9.2) names the node a grant reaches. It is authority, not reference — nothing dereferences it — and it pays the price a code costs: `pan rename`/`mv` cascade it, `rename-prefix` with them, and `pan validate` reports a header naming a dead code (§10.1–§10.2). That is the whole of the exception; do not read it as licence for a second one.
- **No cross-core imports** (I5) and **no second contract** — the CLI JSON is the only boundary (I4).
- **Nothing consumes a lens.** A lens is a terminus: no arrow points at one (§4), so the figures it emits down a pipe answer a hand — yours or an LLM's — never another program. Do not build on top of one; what a lens folds it folds from the cores' JSON, and anything else can fold the same JSON the same way (I4, §12).
- **No rule state the engine keeps.** Auspex holds nothing between runs — no debounce signal, no last-run record, no scheduler; idempotence comes from deterministic slugs + upsert (§9.3), and the wakes are hooks and your own `aus run`, nothing else (§9.4). A rule that wants a memory — a debounce, a note of what it proposed — proposes an ordinary Annales log and reads it back (§8.6), which is a record like any other, not a store the engine maintains. Do not give Auspex a state file.
- **No shared libraries.** Static linking only. Rust has no stable ABI, and a `libpantheon.so` would put twelve independently-versioned apps (§15.5) on one spine object and leave none of them standalone — nobody installs the suite to run one tool (§15.4). Where linked versions differ it is the **format** version that keeps their files compatible, and that is the only compatibility surface this suite has. Disk duplication is accepted.
- **No sphere-locking of cores.** Any core may home records at any node the reality test admits (I7, §6.2).
- **One normalization for every typed token** (§5.1) — codes, slugs, kinds, series and rule names, and filenames alike: lowercase, NFC, Unicode alphabetic-or-numeric plus `_`, everything else folded or dropped. §5.1 states the rule and is its only statement; mixed case and stray characters buy nothing but shift-key friction and case-/encoding-sensitivity bugs at the terminal. (Format notation like `YYMMDD` and env vars like `PANTHEON_RULE` are not typed tokens and are exempt.) Do not add a second casing, and do not add a knob for it.
- **A slug is not frozen** — it is the record's name, and `rename` re-slugs it, cascading refs (§5.4, §7.2). This is the one identity token that moves, and it moves *by cascade, not by a reverse index* — the walk finds the refs (§5.0); do not add an index to make it cheaper.
- **No config file, no behavioral settings.** UI, the keymap (not rebindable, §11.1), mutation-confirm behavior (§7.3), pad width (2), and per-core defaults (Album `person`, …) are hardcoded — Pantheon offers no knobs. The only hand-written TOML is `[code]__.toml` (node annotations) and document frontmatter, both annotation, never behavior (§6.6). Do not add a `config.toml` or per-node tool config. The editor is the environment's too (`$VISUAL`/`$EDITOR`, §7.3) — do not add a `PANTHEON_EDITOR`, a per-core `PENSUM_EDITOR`, or an `--editor` flag. **One carve-out exists, and it is reference data a lens reads, never behaviour it tunes:** a per-programme `[code]_curriculum.toml` (§19.3) declaring a university's grading scale and academic calendar — external facts about a school, homed at the programme node they govern (§6.3) and hand-edited like an annotation, so a published studies tree carries its own scales. It turns a grade symbol into the number the GPA needs (§19.4), which no reading of your life holds; it sets no knob on Studium or any tool, and that boundary is the whole of the exception. A file that tuned behaviour would still be forbidden.
- **No thirteenth verb on a core** (§7.3). `add · edit · rename · move · rm · list · get · series · where · schema · help · version` — that, plus the core's own flags, is a core's whole vocabulary. The spine (`pan`, §5.5) and Auspex (`aus`, §9.6) carry their own structural verbs; that is the shape of what they are, not licence for a core to grow one, and a lens owning no records grows fewer still (§12). Opening a file in an editor is `edit` in its editor form, never an `open`; saying where something lives is `get`/`where`/`pan cd`, never a `--print-path`. A new verb is one more thing to remember in twelve binaries, and the hand it would serve is already served by the TTY rule.
- **No YAML.** TOML for everything hand-written, `+++` frontmatter included — one format, and `toml_edit` the one parser that reads it (§6.6, §13).
- **No versioning of bulk content.** The OS and backups carry it (§6.5).
