## 10. `pan` ‚Äî the structural TUI

Bare `pan` opens a TUI. `pan` never touches `data` ‚Äî it works one layer down, on the tree itself: codes, files, refs, node annotations. Two tabs; annotate is an action on the selected node (¬ß10.3).

### 10.1 Tree

Browse the ontology and act on it. Left pane: codes, symbols, labels, child and record counts. Right pane: the selected node's meta, which cores have files there, which rules are scoped there.

| Verb | Does |
|---|---|
| `pan new <parent> <char> <label>` (or `--def <definition>` for a char-less node, ¬ß5.1) | `mkdir` + scaffold the meta dir (and an empty `[code]__.toml` for top layers); accepts a batch of siblings in one plan (I7) |
| `pan rename <code> [--char C] [--label L]` (a definition-prefix node has no char ‚Äî `--def D` re-derives its definition, code, and entity-node slug) | change defining char and/or label |
| `pan rename-prefix <old> <new>` / `pan rename-pattern <from> <to>` | bulk name fix over the subtree ‚Äî rewrite a code-prefix, or substitute a literal across labels/filenames/slugs/series names; a slug or series-name hit cascades its refs (¬ß5.4), and a code-prefix hit cascades the rule headers naming it (¬ß9.2), exactly as `rename` and `mv` do |
| `pan mv <code> --to <parent>` | re-home a node |
| `pan mv-file <file> --to <code>` | re-home one file ‚Äî a misfiled record, or a whole series (¬ß7.2) ‚Äî rewriting its `[code]` prefix |
| `pan rm <code>` | delete ‚Äî refused if the node has children, record files, **or rules**: a rule is neither stratum (¬ß6.5) but it is code you authored and granted, and the last thing that should drop without a word |

`rename`, `mv`, and `rename-prefix` cascade ‚Äî a node's code is its path (I3), so every child directory name and file prefix under the branch, and every rule header naming the code (¬ß9.2), must change. **Stored entity refs do not** ‚Äî they carry no path, so nothing elsewhere in the tree points at the old code (¬ß5.4); this is exactly what location-independence buys. The cascade is therefore contained to the moved branch plus rule headers. **One exception:** renaming a **definition-prefix** node re-derives its entity-node's slug (the slug *is* the definition, ¬ß5.1), so unlike a triple-node rename it also **cascades that entity's `core:slug` refs** (¬ß5.4) ‚Äî the single node-rename that touches refs. Both run as a **planned transaction**:

1. `--dry-run` (default in the TUI) computes the full plan ‚Äî every directory rename, prefix change, and affected rule header, plus warnings ‚Äî and shows it before anything moves.
2. Nothing is written outside the tree ‚Äî there is no cache dir and no in-flight plan file (¬ß18). If the process dies mid-operation, the half-finished rename is diagnosed directly from **the tree's current state** via `pan validate` (¬ß10.2), whose candidate fixes resolve it; there is no automatic resume and nothing to read back. The tree's current state is the only truth (I1): a partial rename is read from what *is*, never from a log of what was intended.
3. Nothing is locked either. ¬ß6.4's lock is a record file's, and ¬ß18 leaves nowhere to keep a bigger one ‚Äî so a core writing a record inside the branch this operation is moving is a race the tools do not close. One hand moves the tree at personal scale, and what a race would leave is what a crash leaves: a state read back from what *is* (¬ß10.2), not a guarantee bought with a lock the system has no room for.

Confirmation follows the hardcoded mutation rule (¬ß7.3): the structural transaction always confirms before it moves anything.

### 10.2 Validate

`pan validate`'s findings, browsable. **Where a finding has exactly one legal correction, `pan` applies it** ‚Äî an uppercase char, a space or `-` in a name, a stray punctuation mark, a non-NFC form all normalize to a single answer (¬ß5.1), so `pan` fixes them directly. This is not a violation of "`pan` doesn't decide": auto means *there is no choice to make*, not *no confirm* ‚Äî the fix is still a mutation and confirms first (¬ß7.3). **Where a finding has a genuine choice, `pan` never picks** ‚Äî it surfaces **candidate commands** to review and run, and stops at "here's what I found and the fixes that would resolve it." Diagnosis is from the tree's current state ‚Äî there is no transaction history to consult (¬ß10.1), so `pan` reads what *is* and offers the corrections that make it consistent, rather than recalling what a past rename intended.

A finding names the inconsistency and its candidate fixes; where more than one correction is plausible, all are shown and the user picks. A `csa_`-prefixed file inside `cso__/`, for instance ‚Äî misfiled, or a rename that missed it? ‚Äî surfaces both `pan mv-file ‚Ä¶ --to csa` (it belongs at `csa`) and `pan rename-prefix csa cso cso` (it belongs here, wrong prefix), and you choose which is true. The tree can't tell you which was *meant*; you can.

Entity refs (`album:mara`) never dangle on an ordinary (triple or numeric) node rename ‚Äî they carry no path, so re-homing or renaming such a node leaves them untouched (I3); the only thing that rename can leave inconsistent is a *filename prefix* under the moved branch. The **one exception is a definition-prefix node**, whose slug *is* its definition (¬ß10.1): renaming it cascades the entity-node's `core:slug` refs like a record rename, so a crash mid-cascade can strand some on the old slug ‚Äî diagnosed by `pan validate` exactly as a crashed rename is (¬ß5.4, ¬ß7.2). Otherwise a ref dangles only when its target record ‚Äî an entity, document, task, or named series ‚Äî is deleted, which `pan validate` reports for the user to fix at the source.

Scope is strictly the tree: every ref resolves; every file sits at a real node with a matching `[code]` prefix; code collisions (reserved-verb and `root` clashes, ¬ß5.5, ¬ß7.3, and a definition-prefix that prefix-shadows a sibling lineage, ¬ß5.3); slug and hand-named-series-name collisions across nodes (soft ‚Äî reported for you to fix at the source, ¬ß5.4); **a directory under a node that is neither its meta dir nor a legal child** (¬ß5.1) ‚Äî the orphan a crashed rename leaves, `csa__/` stranded beneath a node now named `csb`, and the one finding that cannot inherit the fold's filter to see it: the walk that folds records visits node dirs and meta dirs only (¬ß6.3), so the orphan is invisible to it and to every core, and `validate` therefore descends what the fold declines. That is the only walk in the system that does, and ¬ß13's totality is the fold's claim, never this one's; prefix/directory mismatches; a kind token owned by no installed core (soft ‚Äî a typo, or a core not installed, ¬ß5.5); a rule header naming a dead code; a determined-name series stranded at a node its determinant has left ‚Äî a `Series { named: false }` carrying a name (a `balance`) whose co-located entity is gone, which the spine can see because `named` told it the name is a determinant to match (¬ß7.1); ¬ß7.2 moves the two together, but a hand's `mv` can part them; a def-prefix node with a non-def-prefix child (whose code would be the ambiguous concatenation ¬ß5.1 forbids); and more than one slug-less self-record (`[code]__[kind].json`) at one node (leaving `album:<def>` no single answer). The last two are genuine-choice fixes ‚Äî `pan` shows candidates and never auto-applies, since the definition to use, or the node's true kind, is yours to name. A record whose `data` fails its core's schema is out of scope ‚Äî `pan` doesn't inspect `data`.

### 10.3 Annotate

Edit the selected node's `[code]__.toml` (¬ß5.2) ‚Äî symbol, keywords, deity, explanation ‚Äî in place via `toml_edit`, so hand-written comments and ordering survive (the CLI equivalent is `pan annotate`, ¬ß5.5). This is the only hand-written configuration left in the system (UI, confirmation, and core defaults are all hardcoded ‚Äî ¬ß6.6), and it annotates the ontology rather than tuning behavior.
