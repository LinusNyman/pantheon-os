## 12. Lenses

Porticus apps that lead with a **mosaic** of Tessera tiles (§11.2). What makes a lens a lens is **reach**: it is the app that composes *across* cores — reading each one's JSON off `PATH` and deriving the overlap at render, the way a friend's away-days land on a timeline (§8.1, §8.4) — which is exactly what a core's own TUI may never do, however many tiles it drops in over its own readings (§11.2). A lens owns no primitive and holds no data: every figure it shows is folded from the cores' JSON on each data refresh and stored nowhere (I1, §5.0), and it **never originates a write** (I2) — nothing it renders causes a write-back.

A lens may, however, **relay a human-initiated write** — you mark a task done in Atrium, fix a balance in Speculum — by shelling out to the same core verb a hand would type (`pen edit <slug> --done`, §7.2), never by linking a core's lib (I5). The write crosses the JSON boundary like every other (I4) and the core validates it exactly as it validates a typed command; all a lens changes is who asks for the acknowledgement §7.3 requires. Porticus does: a single focused-row change is itself the acknowledgement and relays direct, while a remove or a bulk action opens the Confirm overlay over a computed `--dry-run` and relays carrying its plan token (§11.1, P§7). Every relay carries `-y` — a lens's child writes down a pipe, where a mutation without it exits `5` (§7.3) — so the confirm has to be the TUI's rather than the CLI's; Porticus supplies the modal, never an exemption. The keystroke came through a lens, but the hand is yours — which is exactly what I2 permits and I8 requires.

**A lens is a CLI too** (I4, §4), and what it emits is the present behind its mosaic: its tiles' folds, the figures themselves. Format follows the hand (§7.3) — at a terminal the bare short opens the mosaic, down a pipe it emits those figures as JSON — so an LLM reads what a human sees (I8), and a headless build is the fold without the chrome (§14). It owns no records, so it grows no verbs: the bare short is the whole surface, plus `help` and `version`. Of the universal flags only those naming no record apply — `-h`, `-V`, `-f`, `-C`, `-q`/`-v` — and the rest are usage errors by the rule that already refuses `-c` to Album (§7.3); it reports its crate and format versions like any app, so `pan doctor` sees it (§5.5, §15.5). Picking one figure out of the emitted JSON is the caller's, as with any tool that declares no read flags of its own (§8.7). Nothing consumes a lens in turn — no arrow points at one (§4) — so that JSON answers a hand, never another program.

**Cores are discovered, not required**: a lens reaches whichever core binaries are on `PATH` (I4, I5). A tile whose core isn't installed is simply absent, and so is the relay that would have written to it — the action is unavailable, not broken (P§7). A lens degrades to what it finds, which is what makes installing one app real (§15.5).

- **Atrium** — the home dashboard: the day folded onto one screen.
- **Speculum** — review across any horizon (day → week → month → year), across every core.
- **Studium** — the studies lens (§19); it reaches every core and hosts the figures a study life derives, the **GPA** among them. Nothing stores it (I1): an enrolment is a Fasti span (§8.4) — the course's *period*, open while you are in it and closed when it ends — and the grade is an **Annales fact** (§8.6) referencing that span, a reading you record rather than a field the span carries (Fasti owns placement, not grading, I5). The GPA is a Tessera fold over those grade facts, each weighted by the course's credits and turned into a number by the programme's grading scale (§19.3–§19.4), computed at render and kept nowhere, derived on sight exactly as net worth is (§8.3). A course still open has no grade fact and is simply not in the fold.
